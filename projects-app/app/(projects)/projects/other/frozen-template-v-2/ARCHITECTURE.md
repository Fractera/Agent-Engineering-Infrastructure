# Архитектура

Здесь находится стартовый замороженный шаблон, предназначенный для быстрого создания автоматизации.
Он включает полностью самодостаточный набор компонентов — **без единой зависимости от внешнего источника**.
Папка самодостаточна настолько, что её можно выгрузить как ZIP-архив, вставить в любой другой проект —
и она сразу заработает.

Документ описывает **как устроено** построенное и **что означают термины**. Что должно быть сделано,
законы и запреты — в `REQUIREMENTS.md` (временный рабочий документ, удаляется по завершении).

## Файловая система автоматизации

```
<автоматизация>/
  NODE-TREE-RULES.md  ← закон строительства дерева узлов: виды, порты, связи
  page.tsx            ← страница автоматизации
  _data/              ← ядро: чем автоматизация является
  _components/        ← как она выглядит
  _lib/               ← как она работает
  api/                ← двери маршрута: как в неё стучатся снаружи
```

Четыре папки отвечают на четыре разных вопроса и не пересекаются. Разметка живёт в `_components/`,
работа — в `_lib/`, описание — в `_data/`, входы снаружи — в `api/`.

### `_data/` — ядро

```
_data/
  automation.json         ← ЕДИНСТВЕННЫЙ источник истины: паспорт, граф, компоненты
  automation.schema.ts    ← схемы Zod: форма ядра, описанная один раз; типы выводятся из них
  load.ts                 ← единственная дверь к ядру: читает и проверяет
```

Читать `automation.json` напрямую нельзя — только через `load.ts`, поэтому непроверенные данные внутрь
автоматизации попасть не могут.

Ядро отвечает на четыре вопроса: **паспорт** — кто эта автоматизация, **граф** — из каких узлов и связей
она состоит, **компоненты** — что показывает её страница, **пользовательские кейсы** — ради каких сценариев
владельца она существует. Внутри компонентов вкладка — это **вид** (календарь), а сущность в её массиве —
один конкретный экземпляр этого вида (один календарь); экземпляров может быть сколько угодно.

**Кейсы определяют автоматизацию.** Их описания обязано быть достаточно, чтобы построить её целиком;
разработка без кейсов не начинается. Кейс = `{ cuid, number, text, status }`, статус идёт
`new → in-development → in-use`, номер — то, чем владелец называет кейс вслух.

Всё, что строится — узел, вкладка, сущность, — описано одинаково: **`info`** (что это) + **`status`**
(`in-development` или `materialized`) + **`warnings`** (что агент сказал владельцу) + **`envKeys`** (какие
ключи окружения нужны для работы). Один словарь на всё: один и тот же факт нигде не называется двумя
разными словами. `info` несёт и паспорт — владелец оставляет инструкцию всей автоматизации, а не только её
частям.

**`info` принимает ровно одну из двух форм** — `{ crudUser }` (слова самого владельца, как пришли) либо
`{ aiSummary }` (краткое описание того, что построила модель). Это **объединение, а не два поля**: два поля
могли бы оказаться заполненными одновременно, и тогда невозможно сказать, какое из них описывает вещь.
Статус сборки — отдельное поле: `info` отвечает на вопрос «что это», `status` — «стоит ли оно уже».

### Граф: три группы узлов и рёбра

Узлы не лежат плоским списком — `graph.nodes.groups` держит **входную**, **срединную** и **выходную**
группы, и каждая объявляет правила прямо над своим массивом. Квота двух уровней: у группы — сколько разных
**видов** обязано присутствовать, у каждого вида — свои `deletion`, `addition` и `minNodes`. Рёбра остаются
рядом, на уровне графа, и ссылаются на `cuid` — раскладка по группам их не задевает.

Отсюда важное свойство: **автоматизация рождается с полным инвентарём дверей** — по узлу на каждый канал
(шесть входных, восемь выходных), по одному коннектору с каждой стороны и три срединных вида, и всё это
скрыто. Строитель не придумывает состав, а раскрывает нужное. Числа считаются из самих словарей каналов,
поэтому новый канал сам поднимает квоту.

Смысл такой формы: правило читается там же, где правятся данные, и нарушить его нельзя — схема сверяет
объявленные правила с эталоном, отвергает узел чужого вида, недостачу и лишний коннектор. Таблица правил —
в [`NODE-TREE-RULES.md`](NODE-TREE-RULES.md) §2.1.

### Идентичность в графе

**У каждого узла и каждого ребра свой `cuid` — и ничто другое их не опознаёт.** Ребро ссылается на
`cuid` узлов (`from` / `to`), человеческое имя живёт только в `name`. Имя можно переименовать,
идентичность — нет; поэтому имя никогда не служит ссылкой.

**Состояние на диаграмме** (`state`: `visible` | `hidden`) есть и у узла, и у ребра: скрытая часть графа
остаётся в ядре и просто не показывается.

### Каналы узла (`ioType`) — закрытый словарь

Канал говорит, **откуда работа приходит** и **куда доставляется результат**. Значение ровно одно из списка,
произвольные строки запрещены; единственная открытая дверь — `custom`.

| Входящие | Исходящие |
|---|---|
| `control-panel` — запрос через Пульт запуска | `public-page` — страница собственного сайта |
| `webhook` — внешняя система по HTTP | `dashboard` — строка или таблица дашборда |
| `cron` — тик расписания | `calendar` — событие календаря |
| `public-page` — форма на собственной публичной странице | `analytics` — график аналитики |
| `telegram-bot` — сообщение собственному боту | `map` — маркер на карте |
| `user-telegram-chat` — сообщение из личного чата пользователя | `email` — исходящее письмо |
| `custom` — любой другой источник | `telegram-bot` · `user-telegram-chat` · `custom` |

Канал несут только входные и выходные виды узлов; у срединного узла канала нет вовсе (`ioType: null`).

### Порты: обязательность и доступные соединения

`in` и `out` — объекты из двух фактов: **`state`** (`required` · `optional` · `prohibit` — насколько порт
обязателен) и **`connections`** (список видов узлов, к которым порт вправе подключаться, либо `null` у
запрещённого порта). Оба факта жёстко следуют из вида узла, и связаны между собой: `prohibit` возможен
только вместе с `null`.

```json
"in":  { "state": "prohibit", "connections": null },
"out": { "state": "required", "connections": ["transform"] }
```

Имён и типов данных портов в ядре нет: что узел принимает и что отдаёт, сказано в его функциях
(`accepts` / `returns`) — один факт не должен жить в двух местах.

**Полная таблица связей, законы рёбер и схема каждого вида узла — в [`NODE-TREE-RULES.md`](NODE-TREE-RULES.md)**
в корне папки. Это обязательный к исполнению закон: схема ядра отвергает узел, объявивший свою строку иначе,
и ребро, ведущее туда, куда источнику вести не позволено. Там же — видимость (`hidden` = функция не
исполняется, узел пропускает данные как ребро без логики), жизненный цикл (замороженный шаблон приходит со
всеми узлами `hidden`), обязательный набор из четырёх видов узлов, закон «одна функция на узел» и рабочий
цикл агента.

### Одна функция на узел

Узел несёт ровно одну функцию — `function: { name, summary, accepts, returns }`, не массив. Сложность
выражается количеством узлов: узел, которому «нужны две функции», — это два узла. Имя функции здесь
совпадает с именем файла-функции в `_lib/nodes/`.

### `_lib/` — функции

```
_lib/
  components/       ← работа вкладок; повторяет уровни `_components/`
    calendar/
    analytics/
  nodes/            ← работа узлов; имена функций совпадают с объявленными в узлах
```

**`_lib/components/`** повторяет структуру `_components/`: у каждой вкладки своя папка того же имени.
Компонент вкладки отвечает за то, как она выглядит; функции этой вкладки — за то, что она считает,
готовит и преобразует. Разделение простое: всё, что не разметка, уходит сюда.

**`_lib/nodes/`** уровней не имеет — это плоский набор функций. Имя функции здесь совпадает с именем,
объявленным в узле (`functions[].name` в ядре): узел говорит, ЧТО он делает, эта папка — КАК.

### `api/` — двери маршрута

Одна дверь — одна папка с `route.ts`; адрес двери продолжает адрес автоматизации:
`/projects/<кат>/<слаг>/api/<дверь>`. Дверей пока нет — они появятся, когда будут названы нужные входы.

### `_components/` — исходный код всех компонентов

Здесь лежит исходный код всех компонентов автоматизации. Он **сгруппирован по вкладкам-сущностям**:
одна вкладка — одна папка, и внутри неё лежит всё, что к ней относится, — и административная половина,
и публичная.

```
_components/
  surface.ts              ← понятие поверхности: "public" | "admin" (объявлено один раз)
  index.tsx               ← ОБЩИЙ МАРШРУТИЗАТОР автоматизации: собирает вкладки
  calendar/
    index.tsx             ← маршрутизатор календаря: общая часть вкладки
    public.tsx            ← календарь публичный
    admin.tsx             ← календарь админ
  analytics/
    index.tsx             ← маршрутизатор аналитики
    public.tsx            ← аналитика публичная
    admin.tsx             ← аналитика админ
```

**Три роли файлов внутри вкладки:**

| Файл | Роль |
|---|---|
| `index.tsx` | **маршрутизатор вкладки** — складывает две половины друг под другом |
| `public.tsx` | **публичная половина** — сама вкладка: только чтение, ни одной настройки |
| `admin.tsx` | **административная половина** — настройки вкладки |

**Маршрутизатор не переключает половины, а складывает их.** Публичная половина идёт **сверху**,
административная — **под ней**:

```
Административный слой          Публичный слой
┌───────────────────┐          ┌───────────────────┐
│ календарь         │          │ календарь         │
│ (публичная часть) │          │ (публичная часть) │
├───────────────────┤          └───────────────────┘
│ настройки         │
│ календаря (админ) │
└───────────────────┘
```

Владелец видит вкладку целиком: сначала сам календарь, под ним — его настройки. Посетитель получает
только верхнюю половину. То есть **в маршрутизаторе лежат обе, а публичная зона забирает одну**.

Отсюда и правило разделения: публичная половина — это вкладка как таковая, и она обязана быть
самостоятельной (владелец смотрит ровно то же, что и посетитель). Административная — только надстройка
над ней, и без верхней половины смысла не имеет.

**Общий маршрутизатор** (`_components/index.tsx`) собирает вкладки и передаёт каждой одну и ту же
поверхность. Поверхность определяется **один раз — адресом страницы** (публичный адрес → `public`,
кокпит → `admin`) и дальше только передаётся вниз; ни один компонент не вычисляет её заново.

**Почему группировка именно такая.** Вкладка — это одна связная вещь, и обе её половины меняются вместе.
Когда они лежат в одной папке, вкладку можно целиком добавить или убрать одним движением, а забыть одну
из половин при правке невозможно. Разложи их по слоям («все публичные тут, все админские там») — и
каждая правка вкладки станет походом по двум местам.

**Новая вкладка добавляется так же:** папка со своим именем, внутри `index.tsx` + `public.tsx` +
`admin.tsx`, и строка в общем маршрутизаторе. Ничего, кроме этого, менять не нужно.

## Словарь

### Дверь маршрута (API-дверь)

**Дверь маршрута — это точка входа снаружи внутрь автоматизации: адрес, по которому её просят что-то
сделать или что-то отдать.** Не страница (страницу человек смотрит), а служебный вход — в него стучится
программа: браузер страницы, слушатель входящих событий, планировщик.

Двери лежат **внутри папки самой автоматизации**, в подпапке `api/`, и адресуются продолжением её же адреса:

```
/projects/<кат>/<слаг>/api/<дверь>
```

**Зачем они нужны.** У автоматизации две поверхности — публичная страница и кокпит владельца — плюс внешние
источники событий. Если бы каждая ходила своим путём, у автоматизации было бы несколько входов с
расходящимся поведением. Дверь — это **один общий вход**: все зовут один адрес и получают один результат.

**Дверь принадлежит своей автоматизации по рождению.** Она рождается вместе с папкой, знает адрес только
своей автоматизации и не умеет обслуживать чужую. Поэтому дверь уезжает в ZIP-архиве вместе с папкой —
и на новом месте работает по новому адресу.

**Кто может войти.** Дверь проверяет входящего: либо это авторизованный человек, либо служебный вызов
с секретом. Адрес двери публичен — вход в неё нет.
