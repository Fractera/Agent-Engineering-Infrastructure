# Правила строительства дерева узлов

**Закон, обязательный к исполнению.** Любая архитектура автоматизации — новая, изменённая, декомпозированная —
строится ТОЛЬКО по этим правилам. Правила не обсуждаются и не обходятся: узел, нарушивший их, схема ядра
(`_data/automation.schema.ts`) не примет, и автоматизация не запустится.

Читать перед тем, как добавить, удалить или соединить хоть один узел.

---

## 1. Из чего состоит узел

У каждого узла ровно два порта — `in` (вход) и `out` (выход). Порт несёт **два факта, и оба обязательны**:

```
port = {
  state:       "required" | "optional" | "prohibit"   // насколько порт обязателен
  connections: [ вид узла, … ] | null                 // к чему порт вправе подключаться
}
```

- `state: "required"` — порт обязан нести связь. Узел без неё незаконен.
- `state: "optional"` — связь допустима, но не обязательна.
- `state: "prohibit"` — связь невозможна. Тогда и только тогда `connections: null`.
- `connections: ["external"]` — связь с узлом **другой** автоматизации. Ради этого существуют коннекторы.

**Жёсткая связка:** `state: "prohibit"` ⟺ `connections: null`. Запрещённый порт со списком связей и
разрешённый порт с `null` — обе записи схема отвергает.

## 2. Семь видов узлов — схема каждого

Читается так: `←` что входит в узел, `→` куда он выходит. `∅` = порт запрещён (`null`).

```
input-connector                          ← вход опционален, выход обязателен
   external ──optional──▶ [ input-connector ] ──required──▶ transform

output-connector                         ← вход обязателен, выход опционален
   condition-success ──required──▶ [ output-connector ] ──optional──▶ external

input                                    ← входа нет вовсе
   ∅ ──prohibit──▶ [ input ] ──required──▶ transform

output                                   ← выхода нет вовсе
   condition-success ──required──▶ [ output ] ──prohibit──▶ ∅

transform                                ← сквозной рабочий узел
   input | transform | condition-success ──required──▶ [ transform ]
   [ transform ] ──required──▶ transform | condition-success | condition-failure

condition-success                        ← ветка успеха
   transform ──required──▶ [ condition-success ] ──required──▶ transform | output

condition-failure                        ← ветка отказа, поток здесь кончается
   transform ──required──▶ [ condition-failure ] ──prohibit──▶ ∅
```

Та же таблица целиком:

| Вид | `in.state` | `in.connections` | `out.state` | `out.connections` |
|---|---|---|---|---|
| `input-connector` | optional | `external` | required | `transform` |
| `output-connector` | required | `condition-success` | optional | `external` |
| `input` | prohibit | `null` | required | `transform` |
| `output` | required | `condition-success` | prohibit | `null` |
| `transform` | required | `input` · `transform` · `condition-success` | required | `transform` · `condition-success` · `condition-failure` |
| `condition-success` | required | `transform` | required | `transform` · `output` |
| `condition-failure` | required | `transform` | prohibit | `null` |

**Узел обязан объявить свою строку дословно** — те же значения, в том же порядке. Ядро проговаривает закон
вслух; иначе сказать его нельзя.

## 2.1. Группы узлов — где какой вид живёт

Узлы не лежат кучей: `graph.nodes.groups` держит **три группы**, и каждая объявляет свои правила прямо над
массивом, которым правит. Правило рядом с данными читается локально — не нужно держать в голове всю таблицу
видов, чтобы понять, что можно положить сюда.

```
graph
 ├─ nodes.groups
 │    ├─ input   { minKinds, kinds: { <вид>: {deletion, addition, minNodes} }, nodes[] }
 │    ├─ middle  { … }
 │    └─ output  { … }
 └─ edges[]
```

**Квота двух уровней** — потому что один уровень считает не то: «не меньше двух узлов» выполняется двумя
`input` без единого коннектора.

```
у ГРУППЫ  minKinds  — сколько РАЗНЫХ видов обязано присутствовать
у ВИДА    deletion  — можно ли удалить узел этого вида
          addition  — можно ли добавить ещё один
          minNodes  — сколько узлов этого вида должно быть
```

Ключи `kinds` **и есть** список разрешённых видов — второго списка, способного разойтись с первым, нет.

| Группа | `minKinds` | Вид | `deletion` | `addition` | `minNodes` |
|---|---|---|---|---|---|
| `input` | 2 | `input` | forbidden | allowed | **6** |
| | | `input-connector` | forbidden | forbidden | 1 |
| `middle` | 3 | `transform` | allowed | allowed | 1 |
| | | `condition-success` | allowed | allowed | 1 |
| | | `condition-failure` | allowed | allowed | 1 |
| `output` | 2 | `output` | forbidden | allowed | **8** |
| | | `output-connector` | forbidden | forbidden | 1 |

**Группа объявляет свою строку дословно** — схема сверяет её с эталоном и отвергает переписанный закон.
Узел чужого вида в группу не принимается.

**Откуда числа** (это не произвольные значения):

```
input  = 6   по узлу на каждый канал словаря, кроме custom
output = 8   по узлу на каждый канал словаря, кроме custom
             (custom — открытая дверь; узел под неё создаётся по надобности)
коннекторы = 1  ровно один: удаление И добавление запрещены
middle = 1 на вид  пол; расти середина может свободно — там декомпозируется работа
```

Числа 6 и 8 считаются **из самих словарей каналов**, а не вписаны руками: новый канал сам поднимает квоту.

**Машинный смысл правил:**

```
deletion: "forbidden"  ⇒ количество узлов этого вида не опускается ниже minNodes
addition: "forbidden"  ⇒ и не поднимается выше — ровно minNodes, ни больше
minKinds               ⇒ в группе представлено не меньше указанного числа разных видов
```

**Следствие для строителя:** автоматизация **рождается с полным инвентарём дверей** — все каналы уже
объявлены и скрыты. Новый входной или выходной узел не создаётся, а **раскрывается** (`hidden → visible`).
Ненужная дверь прячется, а не удаляется — иначе автоматизация потеряет способность войти в групповую.

## 3. Законы связи

1. **Закон источника.** Ребро `A → B` законно, когда вид `B` назван в `A.out.connections`. Проверка идёт
   со стороны источника.
2. **Закон запрета.** Из узла с `out.state: "prohibit"` ребро не выходит; в узел с `in.state: "prohibit"`
   ребро не входит. Исключений нет.
3. **Закон обязательности.** Порт с `state: "required"` обязан нести хотя бы одно ребро внутри графа.
   Порт с `state: "optional"` может стоять несоединённым — так и живут коннекторы, чей второй конец
   находится в чужой автоматизации, вне этого графа.
4. **Закон завершённости.** В графе обязан быть узел, принимающий работу (`input` или `input-connector`),
   и узел, отдающий результат (`output` или `output-connector`).
5. **Закон идентичности.** Узел и ребро опознаются только по `cuid`. Ребро ссылается на `cuid` узлов;
   имя (`name`) — для человека и ссылкой не служит. Петля `A → A` запрещена.

## 3.1. Что несёт каждая строимая сущность

Узел, вкладка и сущность вкладки описываются **одним словарём** — иначе один факт получил бы два названия:

```
info     : { crudUser } | { aiSummary }      // ровно одна форма, вместе они невозможны
status   : "in-development" | "materialized"
warnings : [ { cuid, text } ]                // §11
envKeys  : [ { name, status, comment } ]     // §12
```

`info` несёт и **паспорт**: владелец оставляет инструкцию не только узлу или компоненту, но и всей
автоматизации целиком.

---

## 4. Прямые следствия — то, что чаще всего строят неверно

- **`transform` не выходит напрямую в `output`.** Между ними всегда стоит `condition-success`: результат
  доставляется только по ветке успеха. Цепочка потока минимально выглядит так:

  ```
  input ─▶ transform ─▶ condition-success ─▶ output
  ```

- **Отказ — тупик.** `condition-failure` ничего не отдаёт дальше: поток на нём кончается.
- **Ветвление** выражается двумя рёбрами из одного `transform` — в `condition-success` и в
  `condition-failure`.
- **`input` не имеет входа, `output` не имеет выхода.** Ни `""`, ни `[]`, ни `{}` — только `null`.
- **Коннекторы — единственная дверь наружу.** Ни один другой вид не вправе назвать `external`.

---

## 5. Каналы (`ioType`) — закрытый словарь

Канал говорит, откуда работа приходит и куда доставляется результат. Ровно одно значение из своего списка;
произвольные строки запрещены, единственная открытая дверь — `custom`.

- **Входные виды** (`input`, `input-connector`): `control-panel` · `webhook` · `cron` · `public-page` ·
  `telegram-bot` · `user-telegram-chat` · `custom`.
- **Выходные виды** (`output`, `output-connector`): `public-page` · `dashboard` · `calendar` · `analytics` ·
  `map` · `email` · `telegram-bot` · `user-telegram-chat` · `custom`.
- **Срединные виды** (`transform`, `condition-success`, `condition-failure`): канала нет — `ioType: null`,
  никогда не строка.

---

## 6. Видимость: `visible` | `hidden`

```
node.state = "visible"  ⇒ узел работает: его функция исполняется
node.state = "hidden"   ⇒ функция НЕ исполняется; узел пропускает данные сквозь себя,
                          как ребро без логики
edge.state = "visible"  ⇒ ТОЛЬКО когда видимы оба конца
```

Законы обязательных связей (§3.3) применяются **только к видимым узлам**. Скрытый узел вправе стоять
несоединённым — иначе замороженный шаблон, где скрыто всё, сам себя не прошёл бы.

## 7. Жизненный цикл автоматизации

Живёт **только в паспорте** (`passport.lifecycle`) и правит всем графом:

```
"frozen-template"  ⇒ ВСЕ узлы hidden, все рёбра hidden, ничего не исполняется
                     (это состояние читает интерфейс, чтобы держать элементы заблокированными)
"real-project"     ⇒ ≥1 узел visible  И  ≥1 пользовательский кейс
```

Переход происходит **после первой итерации разработки**: `frozen-template → real-project`, и нужные узлы
переводятся `hidden → visible`. Обратного автоматического перехода нет.

---

## 8. Обязательный набор узлов

Живёт в правилах групп (§2.1) и **больше нигде**: квоты видов означают, что автоматизация всегда несёт
полный инвентарь дверей — шесть входных каналов, восемь выходных, по одному коннектору с каждой стороны и
три срединных вида. Ненужное прячется, а не удаляется. Коннекторы — единственная дверь наружу: только они
вправе назвать `external`.

---

## 9. Одна функция на узел

```
node.function = { name, summary, accepts, returns }   // ОДИН объект, не массив
```

Узел несёт **ровно одну** функцию. Сложность выражается **количеством узлов**, а не количеством функций
внутри узла: узел, которому «нужны две функции», — это два узла.

---

## 10. Декомпозиция середины — главная работа строителя

Основная работа автоматизации живёт в срединных видах: `transform`, `condition-success`,
`condition-failure`. Закон декомпозиции:

```
1 узел = 1 функция = 1 логический шаг
к одному transform может подключаться МНОЖЕСТВО условных узлов
условие НЕ обязано быть булевым — это свойство, а не «да/нет»
```

Примеры множественного ветвления от одного `transform`:

```
transform «оценить показатель»
   ├─ condition-success  «> 25»
   └─ condition-failure  «≤ 25»

transform «определить цвет»
   ├─ condition-success  «цвет = зелёный»
   └─ condition-failure  «цвет ≠ зелёный»
```

Задача строителя — **максимально декомпозировать логический процесс**: мелкие шаги, каждый в своём узле.

---

## 11. Warning — канал сообщений агента

```
warnings: [ { cuid, text } ]        // у КАЖДОГО узла, вкладки и сущности вкладки
[]  ⇒ сказать нечего
```

Агент пишет сюда вместо того, чтобы догадываться. Отдельная обязанность:

```
если в одной автоматизации скапливаются ПРИНЦИПИАЛЬНО РАЗНЫЕ задачи
   ⇒ агент ОБЯЗАН предложить ГРУППОВУЮ автоматизацию сообщением warning
   ⇒ и НЕ строить их все в одной
```

Автоматизации связываются друг с другом через коннекторы (§8) — именно так собирается группа.

---

## 12. Ключи окружения

```
envKeys: [ { name, status, comment } ]   // у КАЖДОГО узла, вкладки и сущности вкладки
name    : UPPER_SNAKE_CASE
status  : "present" — ключ есть и работает
          "missing" — ключа нет вовсе
          "error"   — ключ есть, но не работает (истёк, опечатка, кончились токены)
comment : обязателен при "error" — ИМЕННО его читает агент на следующем запуске
```

Цикл: при генерации кода агент называет ключ, которым пользуется, и ставит статус по факту. На следующем
запуске он перечитывает `status` и `comment`, проверяет ключ снова и обновляет оба поля.

---

## 13. Пользовательские кейсы — четвёртый корневой объект

```
useCases: [ { cuid, number, text, status } ]
status: "new" → "in-development" → "in-use"
```

**Автоматизация определяется своими кейсами.** Их описания обязано быть достаточно, чтобы построить
автоматизацию целиком. `number` — то, чем владелец называет кейс вслух («в кейсе 02 поменяй…»), поэтому
номера уникальны. Разработка без кейсов не начинается; реальный проект без единого кейса незаконен.

---

## 14. Рабочий цикл агента

```
1. прочитать ВЕСЬ automation.json (паспорт, граф, компоненты, кейсы)
2. найти объекты с info: { crudUser }        ← сырая инструкция владельца
3. выполнить работу по этой инструкции
4. заменить на info: { aiSummary }           ← что построено, имя функции и как она работает
5. status: "in-development" → "materialized"
6. не смог — не гадать: написать warning на этом объекте и идти дальше
```

Один узел = одна функция (§9). Объект закрывается по одному, а не пачкой.

## 15. Системная инструкция — приколоченная строка

У каждой сущности ядра есть **системная инструкция**: короткий авторитетный текст о том, что от этой
сущности требуется, читаемый там же, где читается сама сущность. Она заменяет длинный документ, который
слабая модель не дочитывает.

```
systemInstruction: <строка, приколоченная к эталону>
эталон живёт в SYSTEM_INSTRUCTIONS (automation.schema.ts) и БОЛЬШЕ НИГДЕ
модель, изменившая её в automation.json ⇒ ОТКАЗ валидации с именем поля
```

**Где стоит** (18 мест): паспорт · Fractera Pro · граф · объект узлов (для всех узлов) · каждая из трёх
групп · каждый из семи видов внутри своей группы · компоненты · каждая вкладка · кейсы · история.

Правка инструкции — осознанное действие в файле схемы, а не побочный эффект генерации. Строки пока пусты:
владелец пишет их по одной, и каждая забирает на себя кусок этого документа.

---

## 16. Пятый корневой объект — история

```
history: { systemInstruction, versions: [ { cuid, number, createdAt, objectsTouched, summary } ] }
createdAt      : dd-mm-yyyy hh:mm:ss
objectsTouched : сколько узлов и компонентов было создано, обновлено или удалено (число)
summary        : до 500 символов — что сделано в этой версии
```

Версия = один круг работы над автоматизацией.

---

## 17. Паспорт: автор, шаринг, конфликт кейсов

```
passport.author   : идентификатор пользователя, создавшего автоматизацию
                    (у реального проекта обязателен, у нетронутого шаблона пуст)
passport.sharing  : "private" | "public" — публикация в общее пространство не случается сама
useCases.warnings : конфликт кейсов, делающий разработку невозможной
```

Конфликт не принадлежит ни одному кейсу в отдельности (каждый сам по себе непротиворечив), поэтому живёт
на уровне всего набора.

---

## 18. Известная асимметрия таблицы

Две строки не сходятся навстречу друг другу, и это оставлено намеренно:

1. `input-connector.out` называет `transform`, но `transform.in` не называет `input-connector`;
2. `output-connector.in` называет `condition-success`, но `condition-success.out` не называет
   `output-connector`.

Поэтому ребро проверяется **только со стороны источника**. Прежде чем включать проверку с обоих концов,
эти две строки нужно замкнуть — иначе связи коннекторов будут отвергнуты дальним концом.
