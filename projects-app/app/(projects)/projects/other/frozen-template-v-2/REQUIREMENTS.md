# Требования

> **Временный документ.** Он существует только на время нашей работы над шаблоном: сюда постепенно
> добавляются новые секции требований, выполненное отмечается как завершённое. По завершении разработки
> этот файл удаляется — в готовом шаблоне его быть не должно.
>
> Отметки статуса: `не начато` · `в работе` · `завершено`.

---

## 🔴 КРИТИЧЕСКОЕ ПРАВИЛО. Ни одного двойного источника истины

**Документ и всё, что по нему проектируется, строятся так, чтобы двойного источника истины не
существовало.** Это правило действует над всеми требованиями сразу и не имеет исключений.

**Каждый факт живёт ровно в одном месте.** Всё остальное либо ссылается на это место, либо выводится
из него автоматически. Двух мест, где один и тот же факт записан руками, быть не может — даже если
второе «просто для типов», «просто для проверки», «просто для удобства».

**Проверочный вопрос к любому решению:** если эти два места разойдутся, какое из них правда? Если ответ
не мгновенный и очевидный — это уже двойной источник, решение отвергается. Если ответ «второе
пересоздаётся из первого» — это не источник, а производная, и она никогда не правится руками.

**Формулировка правила для проектирования:** сначала называем единственное место факта, и только потом
решаем, как остальные части его читают. Никогда наоборот.

Двойная правда — это то, что ломало архитектуру раньше всего остального: расхождение накапливается
молча, а обнаруживается как необъяснимое поведение («на диске одно, снаружи другое»).

## ✍️ ПРАВИЛО. Язык комментариев

**Шапка — по-русски, тело — по-английски.**

- **Шапка** — комментарий над файлом или над компонентом: что это, зачем существует, как связано с
  остальным. Пишется **по-русски**, для владельца.
- **Комментарии внутри тела** компонента или функции — **всегда по-английски**.

```tsx
// МАРШРУТИЗАТОР КАЛЕНДАРЯ — складывает две половины.
export default function Calendar({ surface }: { surface: Surface }) {
  // the admin half is appended below the public one
  return (…);
}
```

Документы (`ARCHITECTURE.md`, `REQUIREMENTS.md`) — по-русски целиком.

## Требование 0 (ОСНОВНОЙ ЗАКОН). Полная самодостаточность автоматизации

**Статус: в работе** — соблюдается с первой строки шаблона, но проверяемым его делает требование 1.

Автоматизация обязана иметь внутри себя **полностью самодостаточный слой всех компонентов и функций**,
не связанный с внешней средой — настолько, что **любой маршрут можно просто развернуть в любом другом
аккаунте, и он тут же начнёт работать.**

Это **базовое требование**, данное в начале разработки. Оно **старше всех остальных и главнее их**: любое
другое требование этого документа исполняется только тем способом, который его не нарушает. Если решение
удобно, но тянет зависимость наружу — решение отвергается, а не «дорабатывается потом».

**Проверяемый признак (единственный, по нему и сверяемся):** папку маршрута выгрузили ZIP-архивом, вставили
в чужой проект — она работает. Ни одного импорта за пределы своей папки; ни одного пути, который существует
только на нашем сервере.

**Как проверять на каждом шаге:** в любом файле маршрута — только относительные импорты внутри папки.
Появился `@/…` или `../../` за границу папки — это нарушение закона 0, а не деталь реализации.

**Мы строим с нуля.** Ничего из ранее существовавших шаблонов сюда не переносится: заимствованный файл
приносит с собой и заимствованные зависимости, то есть прямое нарушение закона 0. Любой компонент
появляется здесь только написанным под этот закон.

### 🚫 ЗАПРЕТ. Паттерны архитектуры v1 — только по явному указанию архитектора

**Без явного указания архитектора пользователя ЗАПРЕЩЕНО использовать паттерны архитектуры v1.**

Запрет действует на любую форму заимствования, а не только на копирование файлов:

- копировать или переписывать файлы, компоненты, двери, типы и утилиты v1;
- воспроизводить его решения по памяти («сделаю как там, только заново»);
- наследовать его соглашения — имена файлов и папок, формы данных, параметры адресов, контракты вызовов;
- ссылаться на v1 как на обоснование («так уже сделано», «там это работало») в коде или документах.

**Разрешает только архитектор и только явно, под конкретный случай.** Молчание — это запрет; отсутствие
возражений разрешением не является. Разрешение на один случай не распространяется на следующий.

**Что делать вместо этого.** Решение проектируется здесь, от требований этого документа. Если кажется, что
без паттерна v1 не обойтись, — это повод остановиться и спросить архитектора, а не взять его тихо.

Состояние на сейчас: в папке лежат `page.tsx` и два документа; импортов наружу — ноль, закон соблюдён.
Первый настоящий экзамен закону — требование 1 (ядро) и требование 4 (двери и адреса).

Этот документ — единственное место законов, требований и запретов. `ARCHITECTURE.md` их не повторяет:
он описывает устройство построенного и значения терминов.

## Требование 1. Ядро автоматизации — единственный файл `_data/automation.json`

**Статус: не начато**

Ядром автоматизации является единственный JSON-файл `_data/automation.json`. Он включает в себя
перечисление **узлов** и **рёбер**, которые полностью определяют:

- **диаграмму автоматизации;**
- **интерфейс страницы** — наличие или отсутствие определённых параметров определяет присутствие либо
  отсутствие элементов пользовательского интерфейса;
- **исполнение автоматизации.**

## Требование 2. Только файлы этого маршрута, обращения к базам данных запрещены

**Статус: не начато**

Все записи, описывающие автоматизацию, фиксируются **внутри файловой системы этого маршрута**.
Обращения к базам данных **запрещены**.

## Требование 3. Два слоя работы с автоматизацией

**Статус: не начато**

Требование 2 относится к ПЕРВОМУ слою. Слои нельзя смешивать.

**Слой 1 — сама автоматизация.** То, из чего она состоит и что обеспечивает её переиспользование между
разными аккаунтами. Живёт **без баз данных**, только файлами этого маршрута. Именно этот слой уезжает
в ZIP-архиве и работает в чужом проекте.

**Слой 2 — данные самой автоматизации.** То, что она накапливает в работе. Хранится в:

- локальной базе данных;
- локальном объектном хранилище;
- LightRAG.

Для **абсолютно каждой записи** этого слоя связь — **многие ко многим**.

Проверочный вопрос при любом сомнении: «это описание автоматизации или её данные?» Описание — в файлы,
данные — в хранилища слоя 2.

## Требование 4. Архитектура адресов

**Статус: не начато**

У одной автоматизации **две поверхности**, и поверхность определяется **адресом**, а не параметром запроса.
`<кат>` — категория, `<слаг>` — имя автоматизации.

| Поверхность | Адрес | Слой | Кто видит |
|---|---|---|---|
| **Публичная страница** | `aifa.dev/projects/<кат>/<слаг>` | приложение (:3000) | посетитель, без панелей настройки |
| **Кокпит владельца** | `projects.aifa.dev/projects/<кат>/<слаг>` | Проекты (:3003) | architect / manager, после авторизации |
| **Двери маршрута (API)** | `/projects/<кат>/<слаг>/api/<дверь>` | тот же маршрут | обе поверхности + слушатель входа |

Что такое дверь маршрута — определение в `ARCHITECTURE.md`, раздел «Словарь → Дверь маршрута (API-дверь)».

Дополнительно:

- **Языковой префикс публичного адреса** — `aifa.dev/<lang>/projects/<кат>/<слаг>` — обязан вести туда же
  (префикс срезается, маршрут тот же).
- **IP-режим (сервер без домена)** — те же пути по портам: `<ip>:3000/projects/<кат>/<слаг>` (публичная) и
  `<ip>:3003/projects/<кат>/<слаг>` (кокпит).
- **Параметра `?view=public` в контракте адресов НЕТ.** Публичную композицию выбирает публичный адрес.
  Флаг не должен появляться ни в ссылках, ни в документах.

**Как есть сегодня (что чинить):** `app/proxy.ts` (Job 0.5, слой :3000) отвечает на `/projects/*` и
`/<lang>/projects/*` редиректом 307 на `projects.aifa.dev/projects/...` с сохранением query — то есть
публичный адрес приводит посетителя на **кокпит** и упирается в авторизацию. Требование считается
выполненным, когда публичный адрес отдаёт публичную страницу сам, не переводя посетителя в слой Проекты.

## Требование 5. Страница работает без сборки проекта

**Статус: не начато**

Изменение файлов маршрута должно быть видно на странице **без пересборки** приложения.

Как есть сегодня: слой Проекты запущен как `npm run start` (production-сборка), поэтому каждая правка
`page.tsx` доходит до браузера только после `npm run build` + `pm2 reload`. Требование выполнено, когда
маршрут шаблона подхватывается в рантайме.

## Требование 6. Валидация всех объектов через Zod

**Статус: не начато**

**Все объекты валидируются схемами Zod.** Не «важные», не «внешние» — все. Объект, попавший внутрь
автоматизации без проверки схемой, считается ошибкой реализации.

Что именно проверяется:

- содержимое файла-ядра при чтении с диска;
- тело каждого запроса, входящего в дверь маршрута;
- вход узла перед исполнением и его результат перед передачей дальше;
- ответ любого внешнего сервиса — до того, как им воспользовались.

Как проверяется:

- **Форма описана схемой ровно один раз.** Тип для редактора не пишется руками, а выводится из схемы
  (`z.infer`) — этого требует критическое правило о единственном источнике истины.
- **Разные виды узлов — размеченным объединением** (`z.discriminatedUnion` по роли узла): у каждого вида
  свои обязательные поля; лишние поля запрещены (`.strict()`), чтобы опечатка не проходила молча.
- **Отказ обучает, а не падает.** Проверка идёт через `safeParse`, ошибка превращается в список проблем
  с точным путём (`nodes[3].params.channel — обязательное поле`) — он показывается владельцу и
  возвращается агенту-кодеру как объяснение, что именно исправить.

**Пакет.** Зависимость `zod` объявлена в слое Проектов (`package.json`, `^4.3.6`) и установлена на сервере
(4.4.3) — доустанавливать ничего не требуется. При переносе папки в чужой проект `zod` должен быть в нём
установлен; это единственная внешняя зависимость, разрешённая владельцем явно.
