# Антипаттерн: PTY cols больше реальной ширины xterm.js — ломает TUI-терминалы

## Что сломалось

Установка `cols: 500` при spawn PTY или `Math.max(cols, 500)` в resize-хэндлере ломает визуальное отображение Claude Code и Gemini CLI. Терминалы открываются дважды, строки перекрываются, текст смешивается.

## Почему именно Claude Code и Gemini

Claude Code и Gemini используют полноценный TUI с ANSI escape-кодами позиционирования курсора (`ESC[<row>;<col>H`, `ESC[<n>C` и т.п.). PTY убеждён что экран 500 колонок. Когда xterm.js отображает на 120 колонках, escape-коды вида «перейти в колонку 350» ссылаются на позицию за пределами видимой области. Xterm.js переносит строку. Следующий вывод начинается не там где ожидает CLI → строки накладываются друг на друга.

Codex, Qwen, Kimi выводят линейный текст без TUI-позиционирования — они не зависят от точной ширины, поэтому их не ломает.

## Как ломали

В `bridges/platforms/server.js` — PTY-сервер:

```js
// НЕЛЬЗЯ — фиксированный cols при spawn:
proc = pty.spawn(shell, [], {
  cols: 500,   // ← BUG
  rows: 24,
  ...
})

// НЕЛЬЗЯ — запрет на уменьшение cols при resize:
else if (msg.type === 'resize' && msg.cols && msg.rows)
  proc.resize(Math.max(Number(msg.cols), 500), Number(msg.rows))  // ← BUG
```

## Правильный вариант

```js
// МОЖНО — cols при spawn: умеренное начальное значение,
// CLI запускается через 800ms (setTimeout), к тому моменту resize уже пришёл
proc = pty.spawn(shell, [], {
  cols: 220,   // безопасно: не фиксируем 500, даём resize перезаписать
  rows: 24,
  ...
})

// МОЖНО — resize всегда принимает реальную ширину браузера:
else if (msg.type === 'resize' && msg.cols && msg.rows)
  proc.resize(Number(msg.cols), Number(msg.rows))   // ← правильно
```

## Связь с задачей парсинга URL

Первопричина попытки поставить cols:500 — Claude Code выводит OAuth URL (~500 символов), PTY с реальной шириной (~120 cols) оборачивает URL через `\r\n`. `rawBufRef` превращает `\r\n` в пробел → regex `\S+` останавливается на пробеле → URL обрезается.

**Правильное решение:** использовать `bufForSearch` (все пробелы удалены → URL реконструируется целиком) с точным regex, а не расширять PTY. Это решение не касается размера PTY и не ломает терминал.
