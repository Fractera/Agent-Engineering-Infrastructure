# Антипаттерн: SQLITE_BUSY при `next build` — гонка миграций между build-воркерами

**Дата:** 2026-06-24. **Слой:** `services/auth` (Next-приложение auth-сервиса, `fractera-auth`).
**Симптом наблюдался:** деплой падает на шаге `build_auth` («Building auth (production)»), exit 1.

## Симптом
```
SqliteError: database is locked   (code: SQLITE_BUSY)
> Build error occurred
Error: Failed to collect page data for /api/auth/[...nextauth]
```
Падает **недетерминированно**: десятки деплоев проходят, потом один падает. Это признак **гонки**, а не
стабильной ошибки.

## Корень
1. `services/auth/lib/auth/auth.ts` строит адаптер NextAuth **на верхнем уровне модуля**:
   `adapter: SqliteAdapter(getDb())`. То есть БД открывается **при импорте** модуля.
2. `next build` на фазе **«Collecting page data using N workers»** импортирует роут
   `/api/auth/[...nextauth]` в **нескольких параллельных воркерах** (по числу CPU).
3. Каждый воркер вызывает `getDb()` → `runMigrations()` → одновременные **записи** в один файл `auth.db`.
4. Параллельные транзакции записи дают **deadlock на повышении блокировки** (SHARED→RESERVED→EXCLUSIVE).
   SQLite в этом случае возвращает `SQLITE_BUSY` **немедленно** — busy-handler намеренно НЕ вызывается,
   чтобы разорвать взаимоблокировку.

## Почему `busy_timeout` + WAL НЕ спасают
Их и пробовали — не помогло. `busy_timeout=5000` заставляет ждать лок только при «обычной» занятости.
При **write-write upgrade deadlock** (два писателя одновременно повышают блокировку) SQLite возвращает
`SQLITE_BUSY` сразу, минуя таймаут. WAL допускает одного писателя, но конкурентные мигрirующие транзакции
всё равно упираются в этот deadlock. → лечить таймаутом бесполезно, надо **убрать конкурентную запись**.

## Фикс (2026-06-24) — приватная in-memory БД на фазе сборки
В `services/auth/lib/db/index.ts`, в `getDb()`:
```ts
if (process.env.NEXT_PHASE === "phase-production-build") {
  _db = new Database(":memory:");   // приватная БД в каждом воркере: ноль файла, ноль миграций
  return _db;
}
// runtime: реальный файл + WAL + миграции (один pm2-процесс)
```
**Почему это безопасно:**
- Роут `/api/auth/[...nextauth]` **динамический** → методы адаптера при сборке НЕ вызываются, нужен только импорт.
- `SqliteAdapter(db)` не трогает БД при конструировании — все `db.prepare(...)` лениво внутри методов.
- Значит при сборке достаточно вернуть *любой* объект `Database`; in-memory у каждого воркера свой → конкуренции нет.
- В рантайме `NEXT_PHASE !== "phase-production-build"` → открывается реальный файл и один раз идут миграции.

## Правило на будущее
- **Не открывать файловую БД и не запускать миграции на фазе `next build`.** Если модуль вынужден держать
  соединение на верхнем уровне (как адаптер NextAuth) — на `NEXT_PHASE === "phase-production-build"` отдавать
  in-memory/заглушку, а реальную инициализацию (файл + миграции) делать только в рантайме.
- Признак проблемы: `SQLITE_BUSY` именно во время «Collecting page data using N workers» = конкурентный доступ
  к одному файлу БД из нескольких build-воркеров.
- Не «лечить» это `busy_timeout`/WAL — против write-write deadlock они бессильны.
